# /add-migration — Safe Schema Migration Workflow

Guided workflow for making safe Drizzle schema changes with rollback scripts and validation.

**Usage:** `/add-migration` (interactive) or `/add-migration <description>` (e.g., `/add-migration add-reminders-table`)

## Process

### Step 1: Pre-Flight Checks

Before any schema work:

1. **Verify clean git state** — warn if there are uncommitted changes in `packages/db/`
2. **Check current branch** — warn if on `main` (should be a feature branch)
3. **Verify DATABASE_URL** — check `.env.local` exists at the repo root (don't read the value, just confirm it's set)

```bash
git status packages/db/
git branch --show-current
test -f .env.local && echo "ENV OK" || echo "MISSING .env.local"
```

### Step 2: Schema Change

If not already done, guide the user through the schema change. Follow the guardrails:

**Safe operations (proceed):**
- Adding a new table
- Adding a nullable column to an existing table
- Adding an index
- Adding a check constraint

**Dangerous operations (require confirmation):**
- Adding a NOT NULL column without a default (will fail on existing rows)
- Renaming a column (breaks existing queries)
- Changing a column type (may lose data)
- Dropping a column (never do this — deprecate first per guardrails)

If the user's change is dangerous, explain the risk and suggest the safe alternative:
- Instead of dropping a column: "Add a deprecation comment, migrate data in app code, drop in a future PR"
- Instead of NOT NULL without default: "Add as nullable first, backfill, then add NOT NULL in a second migration"
- Instead of column rename: "Add new column, migrate data, update app code, drop old column in separate PRs"

### Step 3: Generate Migration

```bash
pnpm --filter=@petforce/db db:generate
```

Read the generated SQL file from `packages/db/drizzle/` and display it to the user for review.

### Step 4: Create Rollback Script

For the generated migration, create a corresponding rollback SQL file at `packages/db/drizzle/rollback-<migration-name>.sql`.

Rollback rules:
- `CREATE TABLE` → `DROP TABLE IF EXISTS`
- `ALTER TABLE ADD COLUMN` → `ALTER TABLE DROP COLUMN IF EXISTS`
- `CREATE INDEX` → `DROP INDEX IF EXISTS`
- `ALTER TABLE ADD CONSTRAINT` → `ALTER TABLE DROP CONSTRAINT IF EXISTS`

Example:
```sql
-- Rollback for 0042_add_reminders_table.sql
-- Generated by /add-migration

DROP TABLE IF EXISTS "reminders" CASCADE;
```

### Step 5: Validation

Run these checks against the generated migration:

1. **SQL syntax check** — parse the migration SQL for obvious syntax issues
2. **Naming conventions** — verify table/column names use snake_case
3. **Foreign keys** — verify all FK references point to existing tables
4. **Indexes** — verify indexes follow the `<table>_<columns>_idx` naming pattern
5. **One concern per migration** — warn if the migration touches multiple unrelated tables

### Step 6: Dry Run (Optional)

Ask the user if they want to apply the migration now:

- **Yes → apply:**
  ```bash
  pnpm --filter=@petforce/db db:push
  ```
  Then verify the apps still compile:
  ```bash
  pnpm build --filter=@petforce/db --filter=@petforce/core --filter=api
  ```

- **No → skip:** Remind them to run `db:push` before testing

### Step 7: Summary

Print a summary:

```
## Migration Summary

- Migration: 0042_add_reminders_table.sql
- Rollback: rollback-0042_add_reminders_table.sql
- Tables affected: reminders (new)
- Columns: 8 (id, household_id, pet_id, title, due_at, completed_at, created_at, updated_at)
- Indexes: 2 (reminders_household_idx, reminders_pet_idx)
- Foreign keys: 2 (households, pets)
- Breaking changes: None (additive only)

Next steps:
1. Apply migration: pnpm --filter=@petforce/db db:push
2. Build to verify: pnpm build
3. Update exports in packages/db/src/index.ts if needed
4. Create the router: /scaffold-router Reminder
```

## Rules

- **One concern per migration PR** — enforced by the guardrails skill
- **Never drop columns** — deprecate first, migrate data, drop in a separate PR
- **Always generate a rollback script** — even for simple additive changes
- **Always review the SQL** — never blindly apply a migration
- This skill writes files only in `packages/db/` (schema, migration, rollback)
- Follow the Drizzle migration checklist from petforce-guardrails
